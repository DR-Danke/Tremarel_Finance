Executive Summary

The meeting presented the complete architecture and phased roadmap for a centralized operational management platform targeting small restaurants. The system is built on four universal entities (Person, Document, Event, Resource) that compose all business functions without rigid modules. MVP Phase 1 covers employee management, task assignment with WhatsApp notifications, document tracking with expiry alerts, and basic inventory. Discussion Points

1. Problem Statement

Small restaurants suffer from operational disorder: unclear task ownership, expired permits, inventory discrepancies, uncontrolled waste, and forgotten payments/taxes. Current tools (WhatsApp, notebooks, memory) are insufficient. POS systems only handle sales; ERPs are expensive and rigid.

2. Core Entity Architecture

The entire system is built on 4 universal entities: Person, Document, Event, and Resource. All business functions are composed from combinations of these entities rather than rigid modules.

3. Event-Driven System Design

The Event entity serves as the system's nervous system. All key flows pass through Events: documents with expiry dates create Events, Events trigger notifications, recurring tasks generate daily Events, and inventory changes register movements.

4. MVP Phase 1 Scope

Employee directory, task lists with date/frequency assignment, daily WhatsApp task notifications, document upload with expiration dates, automatic pre-expiry alerts, and basic inventory with entry/exit movements.

5. Module Composition Pattern

Business functions are entity combinations: Contracts = Person + Document + Event, Permits = Document + Event, Checklists = Event + Person, Inventory = Resource + Event, Payments = Event + Document, and more.

6. Recipe & Cost Management Extension

Phase 2 adds Recipe and Recipe_Item entities. Recipes define theoretical dish costs by linking resources with quantities. Ingredient cost changes trigger automatic recipe cost recalculation.

7. Invoice Scanning & OCR Pipeline

Supplier invoices are photographed and processed via AI/OCR to extract products, quantities, and unit prices. This automatically updates Resource costs and triggers recipe recalculations.

8. Profitability Alerts

Each Recipe tracks current cost and margin. When margin drops below threshold (e.g., 60%), the system creates an alert Event notifying the owner via WhatsApp/push with dish details and margin data.

9. Permit & Compliance Management

Document + Event combinations handle regulatory compliance: food handling, fire department, commerce registration, extinguisher recharges, and health inspections with advance alerts.

10. Technology Stack

PostgreSQL (Supabase), REST/GraphQL API, React/FlutterFlow frontend, Email + WhatsApp API notifications, DocuSign API for digital signatures (Phase 2).

11. Phased Roadmap

Phase 1 CRUD + Notifications + Basic Inventory   Phase 2 Contracts + Payroll + Production   Phase 3 AI: Prediction, Analysis, Agents

Decisions Made
  • The system will be built on 4 universal entities (Person, Document, Event, Resource) rather than rigid modules
  • Event entity serves as the central nervous system — all workflows pass through Events
  • MVP Phase 1 scope: employee management, task assignment with WhatsApp notifications, document tracking with expiry alerts, and basic inventory
  • Recipe and cost management is explicitly deferred to Phase 2 (depends on established inventory, defined recipes, digitized invoices, and minimum operational discipline)
  • No sub-inventories or complex cost centers in Phase 1 — kept deliberately simple
  • Technology stack: PostgreSQL (Supabase), REST/GraphQL API, React/FlutterFlow frontend, WhatsApp + Email notifications
  • The platform is positioned as a “daily operating system for small restaurants” — not a POS or ERP
  • Design principle: “We don’t design functions. We design a system that enables functions.”

        SYS->>DB: Find matching Resource
        SYS->>DB: Update Resource.last_unit_cost
        SYS->>DB: Create Inventory_Movement (entry, purchase)
        SYS->>DB: Find Recipes using this Resource
        loop For each affected Recipe
            SYS->>DB: Recalculate Recipe.current_cost
            SYS->>DB: Update Recipe.margin
            alt Margin below threshold
                SYS->>DB: Create Event (profitability_alert)
                SYS->>U: Send WhatsApp/Push notification
            end
        end
    end
